# AI 에이전트 워크플로우 인사이트 → 현재 프로젝트 적용

> 출처: [AI 에이전트 워크플로우 3가지 패턴 (GeekNews)](https://news.hada.io/topic?id=27461) · Anthropic 프로덕션 경험 기반 정리

---

## 1. 핵심 논점 3가지

| # | 논점 | 요지 |
|---|------|------|
| **1** | **구조가 자율성보다 먼저** | 에이전트가 늘어날수록 워크플로우 설계가 핵심. 잘못된 패턴 → 레이턴시↑, 비용↑, 신뢰성↓. |
| **2** | **패턴 = 자율성의 범위 설계** | 자율성을 없애는 게 아니라, “어디까지 스스로 결정하게 할지”를 패턴으로 제어하는 것. |
| **3** | **단순함 우선, 필요할 때만 업그레이드** | 단일 에이전트로 충분하면 거기서 끝. 순차로 해결되면 순차로. 첫 결과가 기준 충족하면 끝. |

---

## 2. 비즈니스 기회 포착

- **루틴/게임화 서비스**  
  - “순차 워크플로우”처럼 **단계가 분리된 루틴**을 설계하면, 각 단계별 완료 보상·진행률 노출이 자연스럽다.  
  - “평가자-최적화”처럼 **자기 점검 → 수정** 루프를 유저 루틴에 넣으면(예: 오늘 목표 설정 → 저녁에 자기 평가 → 내일 조정), 품질·일관성 메시지를 서비스 차별화 포인트로 쓸 수 있다.

- **개인 브랜드·1인 기업**  
  - 콘텐츠 제작을 **순차**(초안 → 교정 → 포맷팅) 또는 **평가자-최적화**(초안 → 톤/정책 체크 → 수정)로 구조화하면, “체계적인 제작 프로세스”를 브랜드 스토리로 활용할 수 있다.  
  - RSS/유튜브 큐레이션을 **병렬**로(주제·감성·사실성 동시 평가) 설계하면, “다각도 큐레이션”을 개인 브랜드 포지셔닝에 쓸 수 있다.

---

## 3. 철학적 연결

- **행동 심리**  
  - “가장 단순한 것부터 시작”은 **인지 부하 감소**와 맞닿아 있다. 에이전트든 사람이든, 한 번에 한 역할에 집중할 때 정확도와 만족도가 올라간다.  
  - **평가자-최적화** 루프는 “생성과 비평의 분리”다. 인간의 창작 과정(쓰기 → 읽기 → 고치기)을 구조화한 것과 동일한 원리.

- **자기계발**  
  - “필요할 때만 업그레이드”는 **과한 시스템 구축을 막는 원칙**이다. 1인 기업·개인 루틴에서도 “단계 추가는 정말 필요할 때만” 적용하면, 습관 붕괴와 번아웃을 줄일 수 있다.

## 5. 현재 프로젝트(Focus Feed)에 적용

### 현재 상태

- **요약**: `getVideoContext` → 단일 `summarizeWithGemini(SUMMARY_PROMPT)` → 캐시 저장 → **단일 에이전트**
- **인사이트**: 동일 컨텍스트로 별도 `INSIGHT_PROMPT` 1회 호출 → **단일 에이전트**
- **랭킹**: `getSystemGoalPrompt` + `getRankingPrompt`로 피드 아이템에 우선순위 부여 → **단일 에이전트**

→ 이미 **“가장 단순한 것부터”** 원칙에 잘 맞음. 무리하게 멀티스텝을 넣을 필요 없음.

### 적용 옵션 (필요할 때만)

| 목표 | 패턴 | 적용 예시 |
|------|------|-----------|
| **요약 품질 상향** | 평가자-최적화 | 요약 생성 → “3줄·중립·스포일러 없음” 등 기준으로 평가 에이전트 호출 → 기준 미달 시 수정 프롬프트로 재생성 (최대 N회). 중단 조건(최대 반복 + 품질 임계값) 필수. |
| **한 영상 다각도 분석** | 병렬 | 한 `videoId`에 대해 요약 / 인사이트 / 감성·키워드 추출을 **동시** 호출 후 한 UI에 모아 표시. 비용·레이턴시 트레이드오프 있음. |
| **피드 큐레이션 고도화** | 순차 + (선택) 병렬 | RSS+유튜브 수집 → **순차**: 요약(또는 제목·설명만) → 랭킹. 랭킹 단계에서 “목표별 점수”를 **병렬**로 여러 에이전트에 나눠 계산 후 가중 평균. |
| **비용·속도 최적화** | 지금처럼 유지 | 단일 에이전트 유지. 캐시(Supabase)로 중복 호출만 줄이면 됨. |

### 실무 원칙 요약

1. **지금 구조 유지**로 충분한지 먼저 관찰 (사용자 만족도, 이탈, 재요청 여부).
2. 품질 이슈가 나오면 **평가자-최적화**를 “요약” 한 곳에만 제한적으로 도입 (반복 상한·품질 임계값 명시).
3. “한 번에 여러 관점”이 필요해지면 **병렬** 검토 (비용·결과 합침 로직 설계 후).
4. 세 패턴 **조합** 가능: 예) 순차 파이프라인 안에 “랭킹” 단계만 병렬 평가자 여러 명.
