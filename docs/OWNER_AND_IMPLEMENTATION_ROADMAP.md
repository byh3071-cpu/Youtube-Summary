# 주인(본인) 사용 분리 + 구현 로드맵

주인(운영자)이 제한 없이 쓰는 방법과, 로그인·결제·팀·개인 등 **실제로 구현해야 할 것**을 정리한 문서입니다.

---

## 1. 주인(본인)이 쓸 때 어떻게 분리할지

### 1-1. 목표

- **일반 사용자**: Free / Pro / Team 플랜에 따라 제한(요약 횟수, 브리핑, 팀 인원 등) 적용.
- **주인(본인)**: 제한 없이 전 기능 사용(테스트·운영 편의). 결제 대상에서 제외.

### 1-2. 분리 방식 (택 1 또는 조합)

| 방식 | 설명 | 장단점 |
|------|------|--------|
| **A. 환경 변수로 주인 이메일 지정** | `.env`에 `OWNER_EMAIL=본인이메일@gmail.com` 설정. 로그인한 사용자 `user.email`이 이 값과 같으면 **항상 Pro(또는 Owner)로 간주** → 요약/인사이트/브리핑 제한 없음, Pro 전용 기능 사용 가능, 팀 생성·관리도 제한 없음. | 구현 간단. 주인만 한 명 명시. |
| **B. DB에 owner 플래그** | `profiles` 또는 `users` 테이블에 `is_owner`(또는 `role = 'owner'`). 최초 1명만 수동으로 `true` 설정. 서버에서 `is_owner === true`면 제한 bypass. | 주인 변경·여러 명 운영자 대응 가능. DB 마이그레이션 필요. |
| **C. A + B 조합** | 최초에는 `OWNER_EMAIL`만 쓰고, 나중에 `profiles.is_owner` 도입 후 이메일로 설정된 주인을 DB에 반영. | 초기엔 A만 구현해도 됨. |

**추천**: 우선 **A(OWNER_EMAIL)**만 적용.  
- 로그인 후 `getCurrentUserFromCookies` 등에서 `user.email === process.env.OWNER_EMAIL`이면 **플랜 체크 시 항상 “Pro” 또는 “owner”로 처리**.  
- 요약/인사이트/브리핑 제한, Pro 전용 기능, (나중에) 팀 인원 제한 등에서 **owner면 제한 로직을 타지 않도록** 분기.

### 1-3. 코드에서 쓸 때 (개념)

```text
// 서버: 플랜 판단 시
const plan = user.email === process.env.OWNER_EMAIL ? 'owner' : await getUserPlan(user.id);
if (plan === 'owner' || plan === 'pro') {
  // 무제한 요약, Pro 전용 기능 허용
}
```

- **주인**: `plan === 'owner'` → 모든 제한 bypass.  
- **일반 사용자**: `getUserPlan(user.id)`로 DB/결제 연동 플랜( free / pro / team ) 적용.

이렇게 하면 **주인은 로그인만 하면 되고**, 별도 결제·플랜 설정 없이 전 기능 사용 가능합니다.

---

## 2. 로그인 · 결제 · 팀 · 개인 — 무엇이 있고 무엇을 구현해야 하는지

### 2-1. 현재 구현된 것 ✅

| 항목 | 상태 | 비고 |
|------|------|------|
| **로그인** | ✅ 구현됨 | Supabase Auth + Google OAuth. `/auth/callback`, `LoginButton`, `getCurrentUserFromCookies`. |
| **로그인 필요 기능** | ✅ 부분 구현 | 북마크, 커스텀 소스(채널 추가)는 로그인 시에만 저장·조회(user_id). |
| **플랜/과금** | ❌ 없음 | Free/Pro/Team 구분 없음. 제한 없음. |
| **결제 플랫폼** | ❌ 없음 | Stripe 등 미연동. |
| **팀/조직** | ❌ 없음 | 팀 테이블, 멤버, 초대, 팀 북마크 없음. |
| **개인 vs 팀 UI 분리** | ❌ 없음 | “내 공간” vs “팀 공간” 같은 전환 없음. |

→ **정리**: **로그인은 이미 있음.** 결제·플랜·팀·주인 분리는 전부 **추가 구현**이 필요합니다.

### 2-2. 구현 필요 목록 (우선순위 순)

아래는 “로그인·결제·팀·개인” 관점에서 **실제로 손댈 작업**만 정리한 목록입니다.

| 순서 | 구현 항목 | 내용 | 참고 |
|------|-----------|------|------|
| **1** | **주인(owner) 분리** | `OWNER_EMAIL` env + 서버에서 `user.email === OWNER_EMAIL`이면 플랜/제한 bypass. | 위 1-2, 1-3 |
| **2** | **플랜 개념(DB)** | `user_plan` 또는 `subscriptions` 테이블: `user_id`, `plan`(free/pro), `expires_at` 등. 로그인 사용자당 1행. | Free/Pro 분기 기반 |
| **3** | **사용량·제한 로직** | 요약/인사이트/브리핑 호출 시 “일 N회” 등 제한. DB에 `usage` 테이블 또는 Redis. `getUserPlan`이 free면 제한 적용, pro/owner면 통과. | 개인 수익화 #1 |
| **4** | **결제 플랫폼 연동** | Stripe(또는 토스페이먼츠 등): Pro 구독 상품 생성, 결제 성공 시 webhook에서 `user_plan` 갱신. 결제 실패/해지 시 pro → free 전환. | 개인 수익화 #9 |
| **5** | **플레이리스트 user_id** | `playlists`에 `user_id` 추가, 저장/목록 조회 시 로그인 사용자만 본인 것만. | 개인 수익화 #2 |
| **6** | **팀(workspace)·멤버** | 팀 테이블, 팀 멤버 테이블, 초대(이메일/링크). “팀 만들기” / “팀 참여” 플로우. | 팀 #5 |
| **7** | **팀 북마크·팀 브리핑** | 북마크에 `team_id` 또는 팀 전용 북마크. 팀 목표로 브리핑 API 호출. | 팀 #6, #7 |
| **8** | **팀 결제** | 팀 플랜 상품(Stripe), 팀 단위 구독. `team.plan`, 인원 제한. | 팀 #9 |

**로그인**은 이미 있으므로, “로그인 + 주인 분리 + 플랜 + 제한 + 결제” 순으로 가면 됩니다.

### 2-3. 정리: 로그인 vs 결제 vs 팀

- **로그인**: ✅ 있음. 추가로 필요한 건 **주인 분리(OWNER_EMAIL)**와 **플랜/제한을 읽는 로직**입니다.
- **결제 플랫폼**: ❌ 없음. **Stripe(또는 다른 하나) 연동**이 필요하고, “Pro/Team 구독 결제 → DB 플랜 갱신”만 하면 됩니다.
- **팀/기업**: ❌ 없음. **팀·멤버·초대** 먼저, 그다음 **팀 북마크·팀 브리핑**, 마지막에 **팀 결제** 순서가 맞습니다.
- **개인**: 로그인·북마크·커스텀 소스는 있음. **개인 플랜(Free/Pro)**과 **일일 제한**만 넣으면 “개인” 수익화 구조가 됩니다.

---

## 3. 한 페이지 요약

| 구분 | 내용 |
|------|------|
| **주인(본인) 분리** | `.env`에 `OWNER_EMAIL=본인이메일` 넣고, 서버에서 이 이메일이면 플랜/제한 bypass. 별도 결제 없이 전 기능 사용. |
| **로그인** | 이미 구현됨(Supabase + Google). 주인 분리·플랜 체크만 추가하면 됨. |
| **구현 필요** | ① 주인 분리 ② 플랜 DB ③ 사용량/제한 ④ 결제(Stripe 등) ⑤ 플레이리스트 user_id ⑥ 팀·멤버 ⑦ 팀 북마크·브리핑 ⑧ 팀 결제. |

이 순서대로 하면 “주인은 제한 없이 쓰고, 일반 사용자는 로그인·결제·팀으로 분리”할 수 있습니다.
